### 1.2 详细关系说明

#### （1）管理员表 `admins` — 中心管理实体

| 关联表 | 关系类型 | 关联字段 | 业务含义 |
|:---|:---|:---|:---|
| `projects` | 1:N | `admins.id` → `projects.created_by` | 一个管理员可创建多个项目 |
| `attendance_batches` | 1:N | `admins.id` → `attendance_batches.reviewer_id` | 一个管理员可审核多个批次 |
| `operation_logs` | 隐式 N:N | `admins.id` 作为 `operator_id` | 管理员的操作记录 |

#### （2）司机表 `drivers` — 小程序核心用户

| 关联表 | 关系类型 | 关联字段 | 业务含义 |
|:---|:---|:---|:---|
| `attendance_batches` | 1:N | `drivers.id` → `attendance_batches.driver_id` | 一个司机可提交多个考勤批次 |
| `driver_attendance_records` | 1:N | `drivers.id` → `driver_attendance_records.driver_id` | 一个司机每天一条考勤记录 |
| `driver_favorite_workers` | 1:N | `drivers.id` → `driver_favorite_workers.driver_id` | 一个司机可有多个常用工人 |

#### （3）工人表 `workers` — 被管理对象

| 关联表 | 关系类型 | 关联字段 | 业务含义 |
|:---|:---|:---|:---|
| `groups` | N:1 | `workers.group_id` → `groups.id` | 一个工人属于一个组别 |
| `projects` | N:1 | `workers.default_project_id` → `projects.id` | 一个工人可有一个默认项目 |
| `worker_attendance_records` | 1:N | `workers.id` → `worker_attendance_records.worker_id` | 一个工人有多条考勤记录 |
| `driver_favorite_workers` | 1:N | `workers.id` → `driver_favorite_workers.worker_id` | 一个工人可被多个司机收藏 |

#### （4）工地项目表 `projects` — 业务维度

| 关联表 | 关系类型 | 关联字段 | 业务含义 |
|:---|:---|:---|:---|
| `worker_attendance_records` | 1:N | `projects.id` → `worker_attendance_records.project_id` | 一个项目下有多条工人考勤记录 |

> **注意**：项目与批次之间**不直接关联**。项目是在审核时分配到具体的工人考勤记录上的，这符合"一个批次可包含多个工人，且可多选项目统一分配"的业务规则。

#### （5）考勤批次表 `attendance_batches` — 审核主体

| 关联表 | 关系类型 | 关联字段 | 业务含义 |
|:---|:---|:---|:---|
| `drivers` | N:1 | `attendance_batches.driver_id` → `drivers.id` | 批次由司机提交 |
| `admins` | N:1 | `attendance_batches.reviewer_id` → `admins.id` | 批次由管理员审核 |
| `worker_attendance_records` | 1:N | `attendance_batches.id` → `worker_attendance_records.batch_id` | 一个批次包含多条工人考勤记录 |
| `driver_attendance_records` | 1:N（可选）| `attendance_batches.id` → `driver_attendance_records.source_batch_id` | 司机考勤记录可追溯来源批次 |

#### （6）工人考勤记录表 `worker_attendance_records` — 核心数据

| 关联表 | 关系类型 | 关联字段 | 业务含义 |
|:---|:---|:---|:---|
| `attendance_batches` | N:1 | `worker_attendance_records.batch_id` → `attendance_batches.id` | 记录归属批次 |
| `workers` | N:1 | `worker_attendance_records.worker_id` → `workers.id` | 记录归属工人 |
| `projects` | N:1 | `worker_attendance_records.project_id` → `projects.id` | 记录归属项目（审核时确定）|
| `work_types` | N:1 | `worker_attendance_records.work_type_id` → `work_types.id` | 工作类型 |

#### （7）司机考勤记录表 `driver_attendance_records` — 独立核算

| 关联表 | 关系类型 | 关联字段 | 业务含义 |
|:---|:---|:---|:---|
| `drivers` | N:1 | `driver_attendance_records.driver_id` → `drivers.id` | 记录归属司机 |
| `work_types` | N:1 | `driver_attendance_records.work_type_id` → `work_types.id` | 工作类型 |
| `attendance_batches` | N:1（可选）| `driver_attendance_records.source_batch_id` → `attendance_batches.id` | 追溯来源批次 |

> **设计要点**：司机考勤记录与工人考勤记录**分离存储**，原因见下文"关键设计说明"。

#### （8）工作类型表 `work_types` — 字典表

| 关联表 | 关系类型 | 说明 |
|:---|:---|:---|
| `worker_attendance_records` | 1:N | 工作类型被工人考勤引用 |
| `driver_attendance_records` | 1:N | 工作类型被司机考勤引用 |

#### （9）组别表 `groups` — 字典表

| 关联表 | 关系类型 | 说明 |
|:---|:---|:---|
| `workers` | 1:N | 组别被工人引用 |

#### （10）操作日志表 `operation_logs` — 独立实体

| 关联表 | 关系类型 | 说明 |
|:---|:---|:---|
| 无直接外键 | — | 通过 `operator_id` + `operator_type` 隐式关联管理员/司机 |

> **设计要点**：日志表不设置外键约束，避免业务表删除时级联影响日志完整性，保证审计追踪不可丢失。

#### （11）系统配置表 `system_config` — 键值对存储

| 关联表 | 关系类型 | 说明 |
|:---|:---|:---|
| 无 | — | 独立键值对配置，无业务关联 |

#### （12）司机常用工人表 `driver_favorite_workers` — 中间表

| 关联表 | 关系类型 | 说明 |
|:---|:---|:---|
| `drivers` | N:1 | 归属司机 |

> 联合唯一索引 `uk_driver_worker` 确保一个司机不会重复收藏同一个工人。

## 二、关键设计说明

### 2.1 为什么这样设计批次结构？

#### 业务背景

司机每天需要为多名工人上报考勤。如果让司机一条条提交单个工人的考勤，操作繁琐且容易遗漏；如果直接在考勤记录上附加审核状态，又无法体现"一组工人一起提交、一起审核"的业务整体性。

#### 设计方案

采用**"批次-明细"两级结构**：

```
attendance_batches (批次表)
    └── worker_attendance_records (工人考勤记录表，多条)
```

#### 设计理由

| 设计点 | 说明 |
|:---|:---|
| **聚合提交** | 司机一次选择多个工人，统一填写出勤类型、加班时长，减少页面跳转次数，适合中老年司机使用 |
| **统一审核** | 管理员以批次为单位审核，看到"某司机某日提交的一批工人"，审核视角与司机上报视角一致 |
| **状态集中管理** | 批次状态（待审核/已通过/已撤回）统一控制，避免单个工人记录状态不一致 |
| **撤回粒度** | 司机只能撤回整个批次（未审核前），符合"整批提交、整批撤回"的直觉 |
| **修改权限** | 审核前司机可撤回重提；审核后管理员可修改批次内数据，但批次状态保持"已通过" |

#### 为什么不把司机自己的考勤也放入批次？

司机工资计算规则与工人完全不同（司机工资 = 基础日薪 + 加班工资，不按工人数量），且司机"每天一定全勤"。如果把司机考勤混入工人批次，会导致：
- 批次内记录类型混杂，查询复杂
- 司机工资与工人工资结算周期可能不同
- 违反单一职责原则

因此司机考勤记录**独立成表**，但在批次审核通过时**自动生成**对应的司机考勤记录。

### 2.2 为什么不拆审核表？

#### 常见的另一种设计

```sql
-- 不推荐的设计：拆分审核表
attendance_batches      -- 批次基础信息
batch_review_records    -- 审核记录表（审核人、时间、结果）
```

#### 不拆分的原因

| 原因 | 详细说明 |
|:---|:---|
| **业务简单** | 本系统的审核逻辑极简：只有"通过"一种结果，不存在"驳回"、"退回司机"等复杂流转。审核记录就是"谁在什么时候通过了"。 |
| **状态单一** | 批次只有 3 种状态：待审核 → 已通过（或已撤回）。已撤回的批次不会再次进入审核流程，而是司机重新提交新批次。 |
| **减少 JOIN** | 不拆表意味着查询批次时无需 JOIN 审核表，提升列表页查询性能。 |
| **操作直观** | 管理员在批次列表看到状态，点进去直接修改数据并点击"通过"，无需维护额外的审核记录表。 |
| **日志兜底** | 所有审核操作都会记录到 `operation_logs` 表，满足审计需求，无需单独的审核历史表。 |

#### 批次表的审核字段

批次表直接包含以下审核相关字段，足够支撑业务：

```sql
`status` TINYINT NOT NULL DEFAULT 0 COMMENT '批次状态: 0-待审核, 1-已通过, 2-已撤回',
`review_time` DATETIME DEFAULT NULL COMMENT '审核时间',
`reviewer_id` BIGINT DEFAULT NULL COMMENT '审核人ID(管理员ID)',
```

### 2.3 工资如何避免重复计算？

#### 风险场景

如果没有合理设计，工资计算可能出现以下问题：
- 同一笔考勤记录被多次计入工资汇总
- 修改工人基础日薪后，历史考勤工资被错误重算
- 部分结清后，再次查询工资时出现重复

#### 解决方案：审核时固化工资 + 结清标记

采用**"计算结果持久化 + 结清状态标记"**双保险策略：

**（1）审核时计算并写入冗余工资字段**

```sql
-- 工人考勤记录表中的工资冗余字段
`daily_wage` DECIMAL(10,2) NOT NULL DEFAULT 0.00 COMMENT '当日基础工资(元)',
`overtime_wage` DECIMAL(10,2) NOT NULL DEFAULT 0.00 COMMENT '当日加班工资(元)',
`total_wage` DECIMAL(10,2) NOT NULL DEFAULT 0.00 COMMENT '当日总工资(元)',
```

当管理员点击"通过审核"时，系统根据以下规则**一次性计算**并写入：

```java
// 工资计算逻辑（在 Service 层执行）
BigDecimal dailyWage;
if (attendanceType == 1) {        // 半天
    dailyWage = worker.getBaseDailySalary().multiply(new BigDecimal("0.5"));
} else {                          // 全天
    dailyWage = worker.getBaseDailySalary();
}

BigDecimal overtimeWage = worker.getOvertimeHourlyRate()
    .multiply(BigDecimal.valueOf(overtimeHours));

BigDecimal totalWage = dailyWage.add(overtimeWage);

// 写入考勤记录，后续查询直接读取，不再重新计算
record.setDailyWage(dailyWage);
record.setOvertimeWage(overtimeWage);
record.setTotalWage(totalWage);
```

**设计优势**：
- 即使后续修改了工人的 `base_daily_salary`，历史考勤记录的已固化工资**不会变化**，保证历史数据准确性。
- 工资汇总查询时直接对 `total_wage` 求和，无需实时计算，性能极高且不会重复计算。

**（2）结清标记控制结算边界**

```sql
`is_settled` TINYINT NOT NULL DEFAULT 0 COMMENT '是否已结清: 1-已结清, 0-未结清',
`settled_time` DATETIME DEFAULT NULL COMMENT '结清时间',
`settled_by` BIGINT DEFAULT NULL COMMENT '结清操作人ID',
```

- **按年结算**：查询某工人某年度的工资时，求和该年度所有记录的 `total_wage`
- **部分结清**：管理员可以选定一部分考勤记录进行结清（如结清 2026 年 1~3 月），将这些记录的 `is_settled` 设为 1
- **查询未结清**：`WHERE is_settled = 0`，结果绝对不会包含已结清记录，避免重复结算

**（3）工资发放通过日志表记录**

根据业务要求"不单独建工资发放表"，每次结清操作时：

1. 更新 `worker_attendance_records` 的 `is_settled = 1`
2. 在 `operation_logs` 中插入一条记录：
   - `operation_module = 'WAGE'`
   - `operation_type = 'SETTLE'`
   - `operation_desc = '管理员XXX结清工人【李四】2026-01-01至2026-03-31工资，共计 15000.00 元'`
   - `business_id = 工人ID`
   - `request_params = JSON(结清的考勤记录ID列表和金额明细)`

这样，日志表既承担了审计追踪职责，也承担了工资发放记录的职责。

### 2.4 如何保证数据一致性？

#### 一致性挑战点

| 场景 | 风险 |
|:---|:---|
| 司机提交批次时，部分工人当天已有考勤记录 | 违反"一个工人每天只有一条考勤记录"规则 |
| 审核通过时，系统崩溃导致批次状态已改但工资未计算 | 数据不一致 |
| 审核通过后，管理员修改批次内工人数据 | 可能破坏已固化的工资数据 |
| 司机撤回批次 | 需要级联删除或标记失效工人考勤记录 |

#### 保障措施

**（1）数据库唯一索引 — 物理层防重**

```sql
UNIQUE KEY `uk_worker_date` (`worker_id`, `attendance_date`)
```

- 这是最强约束：即使代码有 bug，数据库也会拒绝重复插入。
- 司机提交批次时，先查询所选工人当天是否已有记录（无论属于哪个批次、什么状态），如有则拒绝提交并明确提示。

**（2）Spring 事务控制 — 业务层原子性**

```java
@Transactional(rollbackFor = Exception.class)
public void reviewBatch(Long batchId, BatchReviewRequest request) {
    // 1. 更新批次状态为"已通过"
    // 2. 更新工人考勤记录（项目分配、数据修正）
    // 3. 计算并写入每个工人的工资冗余字段
    // 4. 生成司机当日考勤记录
    // 5. 记录操作日志
    // 以上任何一步失败，全部回滚
}
```

**（3）审核后数据锁定 — 防止误改**

- 批次状态为"已通过"后，司机端**不可撤回**
- 管理后台可以修改批次内的工人考勤数据，但修改后会**自动重新计算并更新工资冗余字段**
- 建议增加"已结清的考勤记录不允许修改"的校验，防止已发工资被篡改

```java
// 修改考勤记录时的校验
WorkerAttendanceRecord record = workerAttendanceRecordMapper.selectById(recordId);
if (record.getIsSettled() == 1) {
    throw new BusinessException("该考勤记录已结清，不允许修改");
}
```

**（4）司机撤回批次的级联处理**

司机只能在"待审核"状态撤回批次。撤回时：

```java
@Transactional(rollbackFor = Exception.class)
public void withdrawBatch(Long batchId, Long driverId) {
    // 1. 校验批次归属和状态
    AttendanceBatch batch = attendanceBatchMapper.selectById(batchId);
    if (!batch.getDriverId().equals(driverId)) {
        throw new BusinessException("无权操作此批次");
    }
    if (batch.getStatus() != BatchStatus.PENDING.getCode()) {
        throw new BusinessException("只有待审核的批次才能撤回");
    }
    
    // 2. 将批次状态更新为"已撤回"
    batch.setStatus(BatchStatus.WITHDRAWN.getCode());
    attendanceBatchMapper.updateById(batch);
    
    // 3. 级联删除该批次下的所有工人考勤记录
    // （因为工人考勤记录此时还未被审核，无历史价值，物理删除即可）
    workerAttendanceRecordMapper.deleteByBatchId(batchId);
    
    // 4. 记录操作日志
    logService.saveLog(...);
}
```

> 注意：此处采用**物理删除**而非逻辑删除，因为待审核的考勤记录尚未产生任何业务影响（工资未计算、未结清）。若后续需要追踪，可通过 `operation_logs` 查到撤回操作。

**（5）司机考勤记录的自动生成与同步**

当管理员审核通过一个批次时，系统需要为该司机生成当天的考勤记录：

```java
// 在 reviewBatch 事务中
DriverAttendanceRecord driverRecord = new DriverAttendanceRecord();
driverRecord.setDriverId(batch.getDriverId());
driverRecord.setAttendanceDate(batch.getBatchDate());
driverRecord.setAttendanceType(2); // 全天
driverRecord.setOvertimeHours(batchDriverOvertime); // 如有加班
driverRecord.setDailyWage(driver.getBaseDailySalary());
driverRecord.setOvertimeWage(
    driver.getOvertimeHourlyRate().multiply(BigDecimal.valueOf(batchDriverOvertime))
);
driverRecord.setTotalWage(driverRecord.getDailyWage().add(driverRecord.getOvertimeWage()));
driverRecord.setSourceBatchId(batch.getId()); // 追溯来源

driverAttendanceRecordMapper.insert(driverRecord);
```

若该司机当天已有考勤记录（例如已经审核过另一个批次），则：
- 更新现有记录的 `source_batch_id` 为最新批次
- 或者累加加班时长（根据实际业务规则确定）

建议业务规则：**一个司机每天只有一个来源批次**，如果司机一天提交多个批次（前一个被撤回后重新提交），以最后一个被审核的批次为准。

## 三、索引设计详解

### 3.1 核心业务索引

| 索引名 | 所在表 | 字段 | 类型 | 用途 |
|:---|:---|:---|:---|:---|
| `uk_worker_date` | `worker_attendance_records` | `worker_id` + `attendance_date` | 唯一索引 | **最强约束**，防止一个工人一天多条考勤 |
| `uk_driver_date` | `driver_attendance_records` | `driver_id` + `attendance_date` | 唯一索引 | 防止一个司机一天多条考勤 |
| `uk_driver_worker` | `driver_favorite_workers` | `driver_id` + `worker_id` | 唯一索引 | 防止重复收藏 |
| `uk_group_name` | `groups` | `group_name` | 唯一索引 | 组别名称唯一 |
| `uk_type_name` | `work_types` | `type_name` | 唯一索引 | 工作类型名称唯一 |
| `uk_config_key` | `system_configs` | `config_key` | 唯一索引 | 配置键唯一 |

### 3.2 高频查询索引

| 索引名 | 所在表 | 字段 | 用途 |
|:---|:---|:---|:---|
| `idx_driver_date` | `attendance_batches` | `driver_id` + `batch_date` | 司机查询自己的批次列表 |
| `idx_status` | `attendance_batches` | `status` | 按状态筛选批次（待审核列表）|
| `idx_batch_id` | `worker_attendance_records` | `batch_id` | 查询批次内所有工人记录 |
| `idx_worker_id` | `worker_attendance_records` | `worker_id` | 查询某工人的所有考勤 |
| `idx_is_settled` | `worker_attendance_records` | `is_settled` | 查询未结清工资记录 |
| `idx_project_id` | `worker_attendance_records` | `project_id` | 按项目统计考勤 |
| `idx_operator` | `operation_logs` | `operator_type` + `operator_id` | 查询某人的操作记录 |
| `idx_business_id` | `operation_logs` | `business_id` | 按业务ID追踪操作 |

### 3.3 索引优化建议

1. **工资汇总查询优化**：查询某工人某时间段未结清工资时，使用复合索引 `(worker_id, attendance_date, is_settled)` 或确保现有单索引能被优化器合并使用。

2. **日志表分区**：`operation_logs` 数据量增长极快，建议按 `created_at` 做范围分区，或定期归档到历史表。

3. **考勤记录归档**：每年结束的考勤记录可考虑归档到历史表（如 `worker_attendance_records_2025`），减轻主表压力。但当前业务数据量（100+工人，每天100+条记录，一年约 3.6万 条）尚不需要分表，单表完全可以支撑。

## 四、数据字典速查

### 4.1 通用状态枚举

| 枚举值 | 含义 | 使用位置 |
|:---|:---|:---|
| `0` | 待审核 / 禁用 / 否 / 离职 / 未结清 | `batches.status`, `work_types.is_active`, `is_skilled_worker`, `is_employed`, `is_settled` |
| `1` | 已通过 / 启用 / 是 / 在职 / 已结清 | `batches.status`, `work_types.is_active`, `is_skilled_worker`, `is_employed`, `is_settled` |
| `2` | 已撤回 / 已结束 | `batches.status`, `projects.status` |

### 4.2 出勤类型

| 值 | 含义 |
|:---|:---|
| `1` | 半天（基础日薪 × 0.5）|
| `2` | 全天（基础日薪 × 1.0）|

### 4.3 操作者类型

| 值 | 含义 |
|:---|:---|
| `1` | 管理员 |
| `2` | 司机 |
